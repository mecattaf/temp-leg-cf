# 00 — Scope: what `app.leger.run` IS and IS NOT

## What it IS

`app.leger.run` is the **Leger vendor-hosted web configurator** — a React + shadcn SPA backed by a Cloudflare Worker that drives the JSON-Schema envelope for `leger.config.yaml` on a customer's Framework Desktop. The research-thread framing is tight: **"Vercel-level config UI for the local AI stack"** running on that one desktop server. Scope is the Podman Quadlet / Caddy / DNS / Postfix / service-config layer — the `leger.config.yaml` envelope and the files it renders into. Nothing below that layer. Precisely:

1. **Schema-driven form renderer.** JSON-Schema envelope (`leger-schema/*.yaml`) + per-service modules (`openwebui.yaml`, `litellm.yaml`, `mail.yaml`, …) + `x-accounts`, `x-docs`, `compat` extensions. RJSF metadata patterns (not full RJSF) feed shadcn field components; form state via React Hook Form + Zod (the pattern already prototyped in `leger-legacy/src/app/form/FormRenderer.tsx`).
2. **Profile selector.** Top-of-form toggle between `leger-smb-baseline` and `leger-smb-hipaa-adjacent`. The profile pre-checks 12 vs 15 of the 17 LP-XX policies, gates/disables certain fields, injects defaults, and flips Nunjucks conditional blocks in the quadlet templates that get rendered server-side (see `03-profile-toggle.md`).
3. **PR-emitter (not file writer).** On "apply", the CF Worker generates the fully rendered Quadlet + chezmoi + Caddy + DNS tree and opens a pull request to the customer's `<customer>/infra` repo via a customer-installed GitHub App. The customer's own Framework Desktop pulls from that repo via chezmoi + bootc. The Worker **never writes to customer hardware directly** and **never stores long-lived customer credentials**.
4. **The only vendor-hosted Leger surface.** Per noon §5a, this is the sole piece of the four-layer model that lives on Leger's infra (CF Workers + KV + D1 + Pages). Everything else — harness, CLI, tenant state, Vaultwarden, secrets at rest — is on the customer's Framework Desktop.
5. **Scoped strictly to the host-config + service-config layer.** The configurator manipulates Quadlets, Caddy routes, DNS entries, Postfix config, chezmoi `.env.tmpl` references, and the `leger.config.yaml` envelope that renders them. It is "Vercel for local AI stack": what Vercel does for a web app's deployment config, `app.leger.run` does for a Framework Desktop's AI-service config.

## What it IS NOT

Explicit non-goals, each corresponding to a real risk of scope-creep:

1. **NOT deployed on the customer's Framework Desktop.** Tom's emphatic point. The harness does not carry `app.leger.run`. The customer never runs the configurator locally. This is the *only* Leger surface that is SaaS.
2. **NOT an OpenWebUI configurator.** `openwebui-config.com` is a separate Leger-Labs tool aimed at the OpenWebUI community; both apps coincidentally use RJSF + shadcn but they are not the same product, not the same schema, not the same target user. Do not conflate.
3. **NOT a multi-tenant SaaS with billing.** No $99/mo plan, no per-tenant D1 tables, no Stripe, no "Hobby / Pro / Enterprise" pricing ladder. The `Hobby` context dropdown currently in `AppLayout.tsx:105-117` gets ripped. Leger's business model is FDE-driven engagements (~3 per year Blueprint 1/2/3 split), not SaaS conversions.
4. **NOT a Beam.cloud runtime controller.** The legacy codebase was built around Beam.cloud pods (`leger-legacy/docs/ARCHITECTURE-OVERVIEW.md:13-23`). Beam is dropped entirely — runtime is the customer's own Podman quadlets on their Framework Desktop, not Beam pods.
5. **NOT a full-scope 370-field OWUI editor.** The legacy `openwebui-config-schema.json` has 370 variables organized into 50+ `x-category` groups (counted from helper-scripts schema). `ch36-openwebui.md` in the Sodimo manual demonstrates that only ~20 of those are load-bearing (the `WEBUI_AUTH`, `ENABLE_LOGIN_FORM`, `DEFAULT_USER_ROLE`, `RAG_EMBEDDING_*`, `RAG_OPENAI_API_BASE_URL`, LiteLLM connection vars). The configurator surfaces a curated subset per-profile, not the whole schema. Target: ~80–120 fields across all services combined.
6. **NOT a React replacement for the CLI.** The `leger` CLI (per the cli-scoper task) stays as a customer-side utility for account pulls + deterministic quadlet management + model pinning. The configurator emits config; the CLI applies it on the box.
7. **NOT an auth provider.** Customer auth for the configurator runs through **their own Google Workspace** via Cloudflare Access at `app.leger.run`. Leger does not operate its own IdP. Test-mode exists (`TestAuthPage.tsx`) for development only.
8. **NOT a documentation host.** `leger.run` (and `leger.run/docs`) is a separate surface for the `leger-docs` policy pack. `app.leger.run` is the *app*; it links out to the docs but does not serve them.
9. **NOT a release-catalog or marketplace storefront.** The `MarketplacePage` currently in the codebase (`src/pages/MarketplacePage.tsx`) is a legacy artifact from the multi-tenant SaaS plan; it stays gated / deprioritized. Model picker (`ModelsPage`, `ProvidersPage`) is in-scope because `leger-schema` needs model IDs; full marketplace UX is not.
10. **NOT an OS-layer tool.** (Research framing #1 — the load-bearing boundary.) `app.leger.run` does not surface `bootc switch`, harness image selection, OS kernel args, `/usr/lib/bootc/kargs.d/` edits, disk partitioning, LUKS setup, LVM layout, `bootc-fetch-apply-updates.timer` cadence, hardware detection (Strix Halo vs NUC 14 Pro vs MS-01), firmware, boot loader, or any other OS-layer concern. Those belong to the **harness team**: bootc image + `ConditionFirstBoot` systemd units + runtime hardware detection (per research RQ7, Bazzite-style one-image-plus-`/usr/libexec/leger-hardware-setup`). The configurator surfaces **only** the Podman Quadlet / Caddy / DNS / Postfix / service-config layer **above** the OS — i.e., the `leger.config.yaml` envelope. **Compatibility gate:** if the customer's currently-running harness image is incompatible with the pinned `compat.leger_release` (mismatched `harness_digest` in the release-manifest tuple), the configurator **refuses to emit the PR** and surfaces a prominent error: "Your harness image is out of date for this Leger release. Harness upgrade is out-of-band — see [leger-docs/harness-upgrade](…). Apply again after upgrading." Harness upgrade is never automatable from the configurator.

## One-sentence mission statement

> `app.leger.run` is the vendor-hosted schema-driven UI that turns a customer's profile selection + per-service configuration into a pull request against their own infrastructure repository, and nothing else.

Every feature added to this app should be justified against that sentence. If it doesn't fit — it belongs in the CLI, the docs, the harness, or gets cut.
