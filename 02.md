# Leger × Cloudflare maximalist investigation — part 2 of 4
## Per-surface deep-dives (adopt now + adopt post-sprint)

One section per surface in the `adopt now` or `adopt post-sprint` columns of the decision matrix. 100–200 words each, covering what the surface does, why it fits Leger's architecture, what it displaces or enhances, and the specific Workers Launchpad bootcamp session it maps to (per the cohort-6 recap in `workers-launchpad-006` and the ELYD-founder interview paraphrased in `noon-ecosystem-support-map.md` §"Application strategy").

Bootcamp-session naming uses Cohort #1/#2 public session names as the naming convention: "product pricing," "scaling sales," "PMF," "company building," plus the "Solutions Architect office hours" weekly thread.

## 1. Secrets Store — adopt now

**What it does.** Account-level secret management with binding-based Worker access. Secrets are encrypted at rest, propagated globally via Quicksilver, and audit-logged (secrets-store-beta, April 2025). In open beta as of March 2026 with one store per account, 100 secrets per account, scoped to Workers. Binding config in `wrangler.toml`:
```toml
[[secrets_store_secrets]]
binding = "GH_APP_PRIVATE_KEY"
store_id = "<store_id>"
secret_name = "leger-configurator-github-app-pk"
```
The Worker accesses via `env.GH_APP_PRIVATE_KEY.get()`; the key never enters Worker source.

**Why it fits Leger.** LP-03 says key rotation is dogfooded on Leger's own credentials. `04-pr-to-infra-output.md` §"Token rotation on Leger side" says the GH App private key rotates on a 365-day cadence — today that's `wrangler secret put GH_APP_PRIVATE_KEY`. Moving to Secrets Store turns a per-Worker secret into an account-level one with audit logs, so when `leger-configurator-worker` and `leger-release-manifest-promoter-worker` both need the same key they bind to one source.

**What it displaces.** Per-Worker secrets (`wrangler secret put`). Not a regression — the old pattern still works — but it's the right pattern for cross-Worker secrets like the cosign signing key (post-sprint) and eventually the Turnstile secret.

**Launchpad bootcamp mapping.** Solutions Architect office hours — Secrets Store's beta status makes it worth a direct "here's how we'd like to use it, anything on the roadmap we should know about?" conversation.

## 2. Turnstile — adopt now

**What it does.** CAPTCHA-replacement widget that runs non-interactive JS challenges (proof-of-work, browser API probing) to distinguish humans from bots without showing a puzzle. Free for unlimited use in Managed mode (turnstile-ga, September 2023). GDPR / CCPA / ePrivacy compliant, WCAG 2.1 AA compliant. 20 widgets per account on the free tier; Enterprise upgrade only needed past that ceiling.

**Why it fits Leger.** `leger.run` will eventually have a "Request a Leger engagement" CTA (or at minimum, a "Connect GitHub App" button on the configurator's first-time install flow). Without Turnstile, that button is an obvious abuse vector — anyone can hit the Worker endpoint that initiates the GitHub App installation flow. Turnstile on the CTA protects `app.leger.run` from drive-by GH-App-install requests (which the Worker would otherwise dutifully process, minting short-lived tokens until hitting a GH rate limit). Two lines of HTML and a Worker-side `/siteverify` call.

**What it displaces.** No existing protection. This is additive.

**Launchpad bootcamp mapping.** "PMF" session — Turnstile adoption on the signup page is the kind of "ship-it-on-day-one" detail that makes the "meaningfully uses Workers" narrative self-evident. Also a Small App Garden / Built on Cloudflare submission detail (it's explicitly listed on `cloudflare.com/plans/developer-platform/`).

## 3. AI Gateway — adopt now

**What it does.** OpenAI-compatible gateway + Universal Endpoint that sits between an application and any of ~14 LLM providers (Workers AI, Anthropic, OpenAI, Google, Replicate, Mistral, DeepSeek, more — listed on ai-gateway/usage/providers). One endpoint (`gateway.ai.cloudflare.com/v1/{account}/{gateway}/compat/chat/completions`) accepts OpenAI SDK calls; provider-specific endpoints preserve native schemas. Features: caching, retries, fallbacks, rate limits, cost tracking, cf-aig-metadata headers for per-user attribution, Logpush integration (ai-gateway/reference/pricing).

**Why it fits Leger.** Per LP-07 ("on-prem LLM") the *primary* path is the on-box LiteLLM quadlet calling llama.cpp. But the "cloud-heavy escalation" path (per `99-consolidated.md` §B.8 "Symmetric application" + Path (c) architecture note) needs somewhere that is *not Anthropic-direct* to route calls through, otherwise Leger ships a long-lived Anthropic token. AI Gateway accepts **per-request** provider tokens in the `Authorization` header, which Leger can mint at the customer's Vaultwarden on-demand — preserving the no-long-lived-vendor-creds doctrine. Every call gets a cf-aig-metadata stamp that the on-box `run_ledger.db` UPSERTs against for counterfactual-cost tracking.

**What it displaces.** The cloud-escalation path in LiteLLM (the on-box `litellm_master_key`-secured proxy). Leger keeps LiteLLM on-box for local routing; AI Gateway is the *vendor side* cloud fallback only.

**Launchpad bootcamp mapping.** "Scaling sales" session — AI Gateway is the cleanest "Leger saves customers money" narrative artifact because counterfactual-cost tracking is a first-class column in `run_ledger`.

## 4. Workers AI — adopt now (via AI Gateway)

**What it does.** Cloudflare's serverless GPU inference catalog, accessed through the same `env.AI.run("@cf/provider/model", {...})` binding or via AI Gateway's `workers-ai` provider slug. Pricing is Neurons: $0.011 per 1,000 Neurons, 10,000 free per day on Workers Paid (workers-ai pricing, 3 weeks ago). The Agents-Week-2026 catalog includes Kimi K2.5 (256k context, tool-calling, structured output — workers AI announcement in ai-platform April 2026) and Llama Guard 3 8B used by AI Gateway's Guardrails.

**Why it fits Leger.** Strictly as the **cloud-heavy escalation target**. Workers AI + AI Gateway keeps inference on the same CF network as the configurator Worker — no cross-cloud hops, no new long-lived tokens. When a customer's local llama.cpp stack needs a frontier model (rare — LP-07 pushes against this), the escalation is Workers AI first, then Anthropic/OpenAI via AI Gateway. The per-Neuron pricing gives a bounded, predictable surcharge that `run_ledger`'s `cost_eur` column can tally directly against `cost_eur_if_cloud`.

**What it displaces.** Anthropic-direct calls from the configurator Worker. Preserves Path (c) (Claude Code optional, never load-bearing) while giving the configurator a meaningful default "talk to an LLM" pathway that lives inside Cloudflare's provider stable.

**Launchpad bootcamp mapping.** "PMF" session + Workers AI Demo Day — Cohort submissions running on Workers AI are explicitly favoured (internal-ai-engineering-stack shows Cloudflare itself uses Workers AI for 7B tokens/day on Kimi, 77% cheaper than mid-tier proprietary).

## 5. Durable Objects (SQLite-backed) — adopt now

**What it does.** Strongly-consistent per-object compute + embedded SQLite. Each DO has a unique name, globally routable, with its own local SQLite database on disk. SQLite-backed DOs went GA (durable-objects release notes, March 2026) and are now available on Workers Free. Storage billing started January 7, 2026 (durable-objects pricing). 10GB per DO, PITR 30 days. The Agents SDK builds DOs per agent session natively (durable-execution docs).

**Why it fits Leger.** The configurator's current D1 layout (per `04-pr-to-infra-output.md`) stores `engagements/:id/draft` — rows that are hot during a user's active form-editing session and cold forever after. D1 charges per-row-read even for strongly-consistent single-tenant reads. A DO per engagement (named by slug) keeps that draft state next to the compute, with zero-latency reads. More importantly, **OAuth device-grant state** (the 60-second mint window Leger uses for GH App install callbacks per `99-consolidated.md` §B.6) is exactly the use-case DOs were designed for — short-lived, strongly-consistent, request-routed state. Moving the GH-App install-ID rotation log into a DO lets the configurator run a PITR rollback on a bad rotation.

**What it displaces.** Some D1 rows (drafts, install-ID rotation events). D1 stays primary for *cross-engagement* queries (e.g., "how many engagements does this FDE have?"), DOs take over per-engagement hot state.

**Launchpad bootcamp mapping.** "Company building" session — DOs are the flagship Workers primitive Cohort selection committees look at first. The "we use DOs for per-engagement ephemeral state" line is a strong signal.

## 6. Analytics Engine — adopt now (thin slice, wired post-sprint)

**What it does.** Workers-write + SQL-query time-series database optimized for high-cardinality blobs (string dimensions) and doubles (numbers). `env.ANALYTICS.writeDataPoint({blobs, doubles, indexes})` is non-blocking — no added latency. Query via SQL API or Grafana. Unlimited cardinality — customer IDs, request IDs, trace IDs all valid blob values (analytics/analytics-engine get-started, January 2026 and analytics engine intro blog October 2025). Cohort-6-era pricing included a generous free tier.

**Why it fits Leger.** The per-customer `run_ledger.db` stays on-box (LP-04: no inbound ports). But Leger-Labs needs an **aggregate** view — "across all opted-in customers, how many total local tokens today? how much cost saved?" — for the `leger.run` dashboard. Analytics Engine is the correct home. The vendor-side `leger-release-manifest-promoter-worker` (or a sibling telemetry Worker) receives opt-in pings from each customer box (outbound-dial via Tunnel, carrying only aggregate totals — no per-call data), writes them as one blob-per-customer data point, then the `leger.run` dashboard queries `SELECT SUM(double1) AS tokens_local FROM leger_telemetry WHERE timestamp > NOW() - INTERVAL '1' DAY GROUP BY blob1`.

**What it displaces.** No existing tool. This is the first vendor-side aggregate telemetry surface. Does NOT violate the "vendor holds nothing runtime-critical" rule because the box's local `run_ledger.db` is the authoritative source — Analytics Engine is a lossy-by-design aggregate mirror only.

**Launchpad bootcamp mapping.** "Product pricing" session — "we measure cost-saved-vs-cloud per customer" is a pricing-conversation anchor. Also Demo Day — a public `leger.run/dashboard` rendered from Analytics Engine is an obvious showcase artifact.

## 7. Workflows — adopt post-sprint (vendor-side only)

**What it does.** Serverless durable-execution engine built on Workers + DOs. Multi-step programs with automatic retries, state persistence, human-in-the-loop `step.waitForEvent()`, sleep up to weeks, pay-only-for-active-CPU pricing. GA April 7, 2025 (workflows-ga-production-ready-durable-execution); re-architected control plane for agent-scale workloads (workflows-v2, 2 days ago). Each instance is a SQLite-backed DO underneath.

**Why it fits Leger.** **Vendor-side only.** The release-manifest promotion pipeline (per `release-fan-out.md` §2c: "How a manifest is authored") has seven ordered steps — read latest tags of 7 downstream repos, compose draft YAML, open PR, human review, cosign-sign on merge, tag `leger-release-manifest`, fan out comment-PRs to each registered customer infra repo. Each step can fail independently; retrying the whole chain is expensive and wrong. Workflows handles this natively — each step becomes `step.do(...)`, the whole run is auditable in the Workflows dashboard, and `step.waitForEvent('manifest-approved')` pauses the run until Tom hits the "approve" button (release-manifest-promoter-worker's dashboard endpoint).

**What it displaces.** A hand-written Worker with hand-written retry loops against the seven-step manifest promotion. Also displaces the need to build out `leger-bot` as a SaaS-of-one for release ceremony (per `99-consolidated.md` §F.4 and `release-fan-out.md` §7.4).

**Critical constraint.** **Workflows does NOT replace `leger bootstrap` / `leger rotate` CLI verbs.** Those live on-box per the survivability matrix — the box must continue working if `app.leger.run` goes dark. Workflows is strictly for vendor-side orchestration that *produces a git commit*, which the customer can still merge manually if the Workflow is down.

**Launchpad bootcamp mapping.** Solutions Architect office hours — Workflows is evolving fast (workflows-v2 just landed); worth an SA conversation about rate limits and idempotency keys for the promotion pipeline.

## 8. Workers Observability (Logs + Real-time + Logpush + Tail Workers) — adopt post-sprint

**What it does.** Four-layer observability for Workers:
- **Workers Logs** — auto-collected logs queryable in CF dashboard's Query Builder, free + paid tiers, head-based sampling knob (workers-logs, February 2026).
- **Real-time logs** — live `wrangler tail` stream.
- **Logpush** — push logs to R2/S3/Datadog/Splunk/New Relic on Workers Paid (logpush docs).
- **Tail Workers** — attach a Worker that receives `tail()` events from another Worker; useful for custom sampling, filtering, or writing to Analytics Engine (tail-workers docs, February 2026).

**Why it fits Leger.** `app.leger.run`'s users are FDEs, not customers — when something breaks during a customer engagement setup, Tom needs the Workers invocation trace quickly. Workers Logs + Query Builder is the minimum-viable debug tooling for a solo-founder SaaS control-plane. Logpush→R2 handles long-term retention (useful when a customer disputes a rotation timestamp six months after the fact). Tail Workers lets Leger write a custom sampler that forwards only `engagement.slug=<slug>` traces to a future per-customer debug endpoint (useful for FDE co-debugging).

**What it displaces.** Nothing — Leger has no observability today on the vendor surface. This is additive.

**Launchpad bootcamp mapping.** "Company building" session — running a one-engineer SaaS control-plane with zero observability is the classic early-startup footgun Cohort reviewers flag. Observability-from-day-one is a signalling detail.

## 9. Browser Run (formerly Browser Rendering) — adopt post-sprint

**What it does.** Headless browser automation as a Workers binding + REST API. Quick Actions (screenshot / PDF / scrape) need zero code. Browser Sessions support Puppeteer / Playwright / CDP / Stagehand. The new `/crawl` endpoint (March 2026, browser-run crawl changelog) takes a starting URL and returns HTML / Markdown / JSON for an entire site — signed agent, respects robots.txt. Workers Free plan caps at 10 min/day; Paid plan is usage-based.

**Why it fits Leger.** Two post-sprint applications:
1. **`leger-docs` build pipeline.** Per-LP evidence screenshots (a screenshot of the `leger doctor --map-row 3` output, a screenshot of the configurator with LP-02's fields visible, etc.) can be auto-generated at docs-build time rather than hand-taken. Keeps the docs in sync with the UI without manual work per release.
2. **Release-artifact capture.** Per `99-consolidated.md` §E Block 5 "Record a screen capture of the end-to-end flow as a release artifact." Browser Run can automate this post-sprint once Leger has a stable smoke-test harness — every release tag → a fresh screencast of the end-to-end flow, pushed to R2 as the tuple's `release_artifact_url`.

**What it displaces.** Manual screenshots (fragile, drift-prone). One-off screen captures.

**Launchpad bootcamp mapping.** Solutions Architect office hours — Browser Run's crawl endpoint is recent (March 2026) and worth asking about for LP-docs-completeness automation use cases.

## 10. Secrets Store (expanded use) — adopt post-sprint

**What it does.** Same surface as §1, but expanded scope.

**Why it fits Leger.** Post-sprint, Leger needs a home for:
- The cosign signing key for `leger-release-manifest` (per `release-fan-out.md` §2b — cosign-signed manifests).
- The DNF repo signing key for the `leger` CLI RPM.
- Per-environment Workers AI / AI Gateway / Anthropic tokens that rotate on LP-03 cadence.

Each of these belongs at the *account* level (used by multiple Workers — the manifest-promoter, the telemetry worker, the docs-build worker), not the per-Worker level. Secrets Store's binding model + audit log is exactly right.

**What it displaces.** File-on-disk cosign keys (current state) + per-Worker `wrangler secret put` for cross-Worker secrets.

**Launchpad bootcamp mapping.** Same as §1 — SA office hours. The cosign + supply-chain story is a good pitch-deck slide angle for Cohort reviewers who care about developer-platform security signalling.
