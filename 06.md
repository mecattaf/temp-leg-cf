# Leger × Cloudflare — customer-enablement investigation
## Part 2 of 3: The six `leger bootstrap` verbs — per-surface detail

This part expands each of the six `leger bootstrap` verbs into its CLI scope, the CF surfaces it wires on the customer's account, the Claude Code steering patterns it supports, and the plan-tier detection logic it applies.

**Design principles across all six verbs:**

- **Idempotent.** Re-running must converge to the same end state without destroying existing resources. Uses the Cloudflare Terraform provider under the hood (registry.terraform.io/providers/cloudflare/cloudflare, 5.x, 1 week ago) with state persisted in `<customer>/infra/terraform.tfstate` (encrypted via age, per LP-13 pattern).
- **Scoped API token only.** The CLI prompts the FDE for a customer API token scoped to exactly the permissions the verb needs. Token is held in-memory for the duration of the run, never written to disk. (No long-lived vendor credential — LP-02.)
- **PR-based output.** Results emit as a diff against the customer's `<customer>/infra` repo — either a `leger.config.yaml` delta (state references) or a `*.tf` file diff (Terraform resources). Customer reviews and merges. After merge, the on-box `leger apply` writes a `run_ledger` row with `surface='leger_bootstrap', change_type=<verb>`.
- **Claude Code steerable.** Every verb exposes a tool via the `leger` MCP Worker: `leger.bootstrap.perimeter.plan`, `.apply`, `.rollback`. Claude Code can chain these with `mcp.cloudflare.com`'s `search()` and `execute()` for combined "configure box + configure cloud" runbooks.
- **Plan-tier gated.** Each verb's `--help` surfaces "requires Workers Paid plan" / "requires Zero Trust Pay-as-you-go" for features outside Free. `leger doctor --cloudflare-features` reports what's unlocked on the customer's account.

## 1. `leger bootstrap perimeter`

**What it wraps:** Tunnel + Access (applications + policies + IdP binding) + DNS + Turnstile.

**What it produces on the customer's CF account:**

- One Cloudflare Tunnel named `leger-<customer-slug>` bound to the customer's DNS zone. `cloudflared` daemon installed on-box (already today, unchanged).
- One Access Application per on-box service, in `self-hosted` mode (developers.cloudflare.com/cloudflare-one/access-controls/applications/http-apps/self-hosted-public-app/, 6 days ago). Defaults: Cockpit, OpenWebUI, LiteLLM, Twenty, Paperclip, Piler, Datasette, Vaultwarden. Each app bound to a specific hostname (e.g., `openwebui.sodimo.eu`), each mapped into the tunnel config, each tagged with labels like `lp:LP-01` for audit grouping.
- Access Policies — one per team role. Defaults: `all-employees` (any `@<customer-domain>` IdP user), `engineering` (Google group `engineering@`), `finance` (Google group `finance@`), `fde` (named FDE emails + Tom's email as break-glass). Policies are **reusable** — attached to multiple applications (per developers.cloudflare.com/cloudflare-one/access-controls/policies/policy-management/, 1 week ago).
- One DNS record per service, CNAME-to-tunnel. All proxied.
- (Optional) One Turnstile widget per customer-run public form. If the customer has a public-facing landing page on their Pages (row 2 of `leger bootstrap dashboards` below), Turnstile widgets gate the contact form / signup form. Free tier caps at 20 widgets per account (developers.cloudflare.com/turnstile/plans/, 1 week ago) — ample for SMB.

**Claude Code steering example:**
```
FDE: "Add a new service Paperclip at paperclip.sodimo.eu, behind Access, allow finance@ and engineering@."

Claude: 
[calls leger.bootstrap.perimeter.plan with --service=paperclip --hostname=paperclip.sodimo.eu 
  --policy=finance,engineering]
[Claude shows the diff: +1 Access application, +1 tunnel route, +1 DNS CNAME, +0 policies (reused)]
[FDE approves]
[Claude calls leger.bootstrap.perimeter.apply]
[Claude calls leger.doctor --service=paperclip to verify]
[run_ledger row written: surface='claude_code', subject='added paperclip service', 
  prompt_hash=<hash of this turn>, change_type='service_add']
```

**Plan-tier notes.** Cloudflare Access Free tier: 50 users, 50 applications, 10GB transfer (developers.cloudflare.com/cloudflare-one/, 1 week ago). Sodimo at 30 people × 8 services = 240 user-app combinations but only 30 distinct users, fine. Customer graduates to Pay-as-you-go ($7/user/month) only if they exceed 50 users or want features Leger doesn't need (CASB / DLP / RBI). Tunnel is free on all plans (no user-count limit).

**Why customers care.** Pre-Leger, a 30-person SMB has no hope of running a SaaS-grade identity perimeter. Post-`leger bootstrap perimeter`, they have Okta-quality ZTNA with full audit logs, on their own domain, for $0/month of CF cost.

## 2. `leger bootstrap dashboards`

**What it wraps:** Pages (migrating to Workers-with-static-assets per developers.cloudflare.com/workers/static-assets/migration-guides/migrate-from-pages/, 1 week ago) + Access + Workers Builds.

**What it produces on the customer's CF account:**

- One Pages project per engagement named `<customer>-dashboards`, connected to the `<customer>/infra` repo. Subdirectory (e.g., `dashboards/`) auto-builds on push.
- Multiple dashboard routes rendered from `run_ledger.db`:
  - `/inventory` — current stock from Twenty synced to box (customer-specific).
  - `/tokens` — per-employee token spend vs counterfactual cost (generic; driven by `run_ledger` + AI Gateway).
  - `/backups` — last successful backup per LP-13; next scheduled; drill history.
  - `/rotations` — service-token rotation evidence per LP-03; next rotation due.
  - `/compliance` — LP-XX pack evidence matrix (matches the `0.1-controls-evidence-map.md` view).
  - `/handoff` — next-FDE onboarding packet (current state of box, latest release, incident log).
- One Access application per dashboard route + one catch-all for the Pages project itself. Policy defaults per dashboard:
  - `/inventory`: `all-employees`
  - `/tokens`: `engineering` (admin) + `all-employees` (read-only — achieved via Worker-side role check on the JWT)
  - `/backups`, `/rotations`, `/handoff`: `fde` only
  - `/compliance`: `fde` + `audit-reviewer` (for when an insurer/auditor temporarily needs read-only access — time-boxed Access session)
- Data flow: each dashboard is a client-side React/SPA that calls a customer-owned Worker (deployed by `leger bootstrap automation`, §6 below) which reaches the box via Tunnel, queries the local Datasette (already quadletted per `05-changelog-architecture.md` §6), and returns JSON. No raw data leaves the box; only the dashboard's view-model does.

**Claude Code steering example:**
```
FDE: "Add a dashboard showing total cost saved vs cloud for Q2 2026."

Claude:
[reads run_ledger schema via leger MCP]
[writes a new dashboard route /savings-q2 as a React component]
[calls leger.bootstrap.dashboards.apply --route=/savings-q2 --policy=all-employees]
[Pages auto-deploys on push to <customer>/infra main]
[Access app is auto-configured via Terraform]
[run_ledger row: surface='claude_code', subject='added savings-q2 dashboard']
```

**Plan-tier notes.** Pages (free forever for static content) or Workers-with-static-assets (also free for static requests per developers.cloudflare.com/workers/platform/pricing/, 1 month ago). Workers Builds is free for git integration. No customer upgrade needed for small dashboards. High-volume dashboards (>100k requests/day) would want Workers Paid ($5/mo) for higher limits.

**Why customers care.** This is the *visible value* of Leger. Michel at Sodimo doesn't care about Cloudflare-the-network; Michel cares that there's a URL he can bookmark that says "this is how the stack is doing." Post-Leger, every team member has a live, role-gated, auditable view into the parts of the operation they own. This dashboard stack is also **the customer's compliance evidence** — when an auditor asks for rotation logs, the FDE grants a time-boxed Access session to `/rotations` and shows the dashboard. No emailed CSVs.

## 3. `leger bootstrap media`

**What it wraps:** R2 (backup sink + optional asset CDN) + DNS + optionally Images.

**What it produces on the customer's CF account:**

- One R2 bucket `<customer>-backup` with object-lifecycle policy: transition to cold storage after 30 days, deletion never. Encrypted at rest (CF default) + client-side age-encrypted (per LP-13). Credentials scoped to write-only for the on-box backup job.
- One R2 bucket `<customer>-assets` (optional) for the tenant-changelog's static assets — screenshots, docs media, etc. Public via R2's custom domain feature.
- DNS CNAME `assets.<customer-domain>` → R2 custom domain.
- (Optional, post-sprint) Cloudflare Images for per-customer image optimization — only if the customer actually hosts a meaningful volume of images (e.g., they run an e-commerce catalog). For most SMB engagements, R2 direct is enough.

**Claude Code steering example:**
```
FDE: "The backup to R2 has been failing for 3 days. Diagnose."

Claude:
[calls mcp.cloudflare.com/execute with a query for R2 bucket status + IAM events]
[reads run_ledger rows for surface='backup' in last 72h]
[finds: token expired on day 1; rotation event didn't update chezmoi; subsequent backups failed auth]
[suggests: leger rotate r2-token]
[FDE approves]
[Claude calls leger rotate r2-token (CLI subcommand) which uses the customer CF API token 
  to mint a new scoped token, updates Vaultwarden, chezmoi applies, backup retries]
```

**Plan-tier notes.** R2 free tier: 10GB storage, 1M Class A ops/month, 10M Class B ops/month (free for Workers Free, included on Workers Paid). Zero egress. Sodimo's `run_ledger` + encrypted file backups are well under 10GB; customer stays on Free forever. Customers with heavier backups (media archives) move to paid R2 ($0.015/GB/month beyond free tier) which is still materially cheaper than every S3 alternative.

**Why customers care.** LP-13 "age-encrypted backup" is a trust-moat feature — the customer's backups are readable only with their Vaultwarden master password, which Leger does not have. R2-on-their-own-account means they own the bucket, they own the encryption keys, and if Leger disappears tomorrow the backup remains recoverable with just their password + an age binary.

## 4. `leger bootstrap identity`

**What it wraps:** Access IdP binding + Access for SaaS + Secrets Store.

**What it produces on the customer's CF account:**

- IdP binding to the customer's Google Workspace (or Microsoft 365 / Okta / Auth0 / Stytch / custom SAML — Cloudflare supports all, per developers.cloudflare.com/cloudflare-one/access-controls/applications/http-apps/, 2 days ago). One-time OAuth flow during bootstrap; Cloudflare stores the IdP refresh token in their Access tenant (not Leger's). Leger never holds IdP credentials.
- Access for SaaS application wrappers (per blog.cloudflare.com/cloudflare-access-for-saas/, October 2025). For each SaaS the customer uses (Twenty CRM, Paperclip, Salesforce, GitHub, 1Password, etc.), Access acts as the SAML broker between their IdP and the SaaS. Benefit: one IdP, one audit log, one off-boarding flow.
- Secrets Store (developers.cloudflare.com/secrets-store/integrations/workers/, March 2026, open beta, 100 secrets/account) for service tokens that the customer's Workers use. Examples:
  - Twenty API token (used by nightly sync Worker — §6 below)
  - Shopify API token (if customer runs e-commerce on Shopify)
  - DocuSign / e-sign tokens for automated invoicing
  - Each scoped to specific Workers via bindings; rotated on LP-03 cadence.

**Claude Code steering example:**
```
FDE: "Onboard Salesforce as a new SaaS app; only engineering@ and sales@ can access."

Claude:
[calls mcp.cloudflare.com/execute to create a new Access for SaaS application for salesforce]
[Claude displays the SSO URLs + cert fingerprint needed for Salesforce-side config]
[FDE configures Salesforce side (human action — can't be fully automated)]
[Claude calls leger.bootstrap.identity.apply --saas=salesforce --policy=engineering,sales]
[run_ledger row: surface='claude_code', subject='added Salesforce SSO']
```

**Plan-tier notes.** Access for SaaS works on Cloudflare for Teams free plan per blog.cloudflare.com/cloudflare-access-for-saas/ ("available to all Cloudflare for Teams customers, including organizations on the free plan"). Secrets Store is open beta, free during beta, limit 100 secrets/account — ample for SMB.

**Why customers care.** Most SMBs have a shadow IdP problem: Google Workspace + three SaaS accounts per employee + no audit trail. Post-`leger bootstrap identity`, they have **one** identity broker (Access) in front of everything. Off-boarding an employee: revoke from Google Workspace, Access immediately blocks all connected SaaS. Audit request: show the auditor the Access logs for the past year, one view. This is an enterprise-grade capability, free on the customer's own CF account.

## 5. `leger bootstrap ai`

**What it wraps:** AI Gateway (per-engagement) + Workers AI (optional escalation) + Analytics Engine (per-employee attribution).

**What it produces on the customer's CF account:**

- One AI Gateway instance named `leger-<customer-slug>` with Universal Endpoint enabled (developers.cloudflare.com/ai-gateway/usage/universal/, January 2026) and OpenAI-compatible `/chat/completions` endpoint (developers.cloudflare.com/ai-gateway/usage/chat-completion/, March 2026). Configured with:
  - Fallback chain: Workers AI (Kimi K2.5 default) → Anthropic → OpenAI.
  - Cache TTL: 3600s default (saves re-inference on repeated prompts).
  - Rate limits: per-user via `cf-aig-metadata` header.
  - Logpush → customer's R2 bucket for long-term retention.
  - Guardrails enabled (uses Llama-Guard-3-8B on Workers AI, per developers.cloudflare.com/ai-gateway/reference/pricing/, 3 days ago).
- On-box LiteLLM configured to point at the Gateway endpoint as its "cloud escalation" provider. Per-request tokens minted on-demand from Vaultwarden (no long-lived vendor creds).
- Analytics Engine dataset `<customer>-ai-metrics` with blobs `{user_id, service, model, provider, status}` and doubles `{tokens_in, tokens_out, cost_eur, cost_eur_if_cloud}`. Fed by every AI Gateway call via a Tail Worker on the Gateway Worker. Queryable via SQL for the `/tokens` dashboard.

**Claude Code steering example:**
```
FDE: "Show me last week's LLM spend by team."

Claude:
[calls leger MCP: query run_ledger for surface='llm_call' in last 7 days, group by employee_team]
[results show local spend (€23) vs what it would have cost in cloud (€187)]
[optionally: calls mcp.cloudflare.com/execute to cross-check with AI Gateway logs over same window]
[returns a rendered table, same shape as the /tokens dashboard]
```

**Plan-tier notes.** AI Gateway is free on all Cloudflare plans (ai-gateway/reference/pricing, 3 days ago: "AI Gateway is available to use on all plans. AI Gateway's core features available today are offered for free"). Workers AI is priced at $0.011 per 1,000 Neurons with 10,000 free/day on Workers Paid (developers.cloudflare.com/workers-ai/platform/pricing/, 3 weeks ago). Analytics Engine has a generous free tier on Workers Paid. Customer typically needs Workers Paid ($5/mo) for this cluster — one of the few places Leger legitimately encourages the upgrade, with receipts (the AI Gateway cached-hit savings typically exceed $5/mo within a week for any customer actually using LLMs).

**Why customers care.** Pre-Leger, an SMB using LLMs has zero visibility into per-employee spend, zero caching, zero fallback if one provider goes down, and gives their Anthropic token to every employee because centralizing it is operationally complex. Post-`leger bootstrap ai`, they have one gateway, one per-employee audit log, automatic caching (typical SMB re-asks the same question 30-40% of the time and AI Gateway caches it for free), automatic fallback across providers, and counterfactual-cost reporting that makes the "we saved $X by running local" story defensible with numbers.

**The counterfactual column is the pitch.** `run_ledger.cost_eur_if_cloud` is computed per-call based on the model the customer *would* have used (from their LiteLLM config) and the token count. AI Gateway's per-provider cost logs cross-verify. The monthly dashboard shows `SUM(cost_eur_if_cloud) - SUM(cost_eur)` and this is the number that justifies the whole Leger engagement to the customer's CFO. No other SaaS has this.

## 6. `leger bootstrap automation`

**What it wraps:** Workers + Workers Builds + Durable Objects + Queues + Workflows (post-sprint) + Workers Observability.

**What it produces on the customer's CF account:**

- A customer-owned `<customer>-automation` Workers project, connected to `<customer>/infra` via Workers Builds (developers.cloudflare.com/workers/ci-cd/builds/, Feb 2026). Auto-deploys on merge.
- **`email_outbox` Queue** (already in use per LP-15) — customer-owned.
- **Dashboard API Worker** — the Worker that the `/dashboards` (§2) SPAs call to fetch `run_ledger` data via Tunnel. Reads-only.
- **Nightly sync Workers** — one per external SaaS the customer integrates with. Examples: Twenty→box sync, Shopify→box sync, DocuSign webhook handler. Each Worker has scoped Secrets Store bindings and runs on a cron trigger.
- **Durable Objects** for realtime state:
  - Live dashboard state (who's currently viewing which dashboard, for collab features).
  - OAuth device-grant nonces during Access login flows.
  - Per-user rate-limit counters for LLM calls (enforced at the LiteLLM→AI-Gateway boundary).
- **Workflows (post-sprint)** for multi-step business processes:
  - Quarterly rotation drill: the 14-step rotation ceremony where every credential gets rotated, every quadlet restarted, every backup verified. Durable execution means it survives step failures and resumes.
  - Invoice-approval chain: PDF uploaded → routed to approver → countersigned → archived to R2.
  - Backup drill: nightly verification of last 7 backups, random restore test, alert if anything fails.
- **Observability** turned on for every Worker: `observability = true` in wrangler config. Logs query via Query Builder in customer's own CF dashboard. Logpush → customer's R2 for retention.

**Claude Code steering example:**
```
FDE: "Set up a weekly backup drill that picks a random backup from the last 7 days, restores 
     it to a scratch directory, verifies checksums, and emails Michel if anything fails."

Claude:
[generates a Workflows definition: step.do('pick-random-backup'), step.do('restore'), 
   step.do('checksum'), step.do('notify-on-fail')]
[pushes to <customer>/infra/workflows/backup-drill.ts]
[Workers Builds deploys on merge]
[creates a cron trigger via mcp.cloudflare.com/execute for Monday 02:00]
[registers a run_ledger trigger: every drill execution writes surface='workflow', 
   subject='backup-drill' row]
```

**Plan-tier notes.** Workers Paid ($5/mo) required for Durable Objects key-value storage, higher Logs limits, Queues beyond free tier, and Workflows. DO SQLite is free on Workers Free. Most customers running any non-trivial automation graduate to Workers Paid; the $5/mo pays for itself the first time the sync Worker saves 30 minutes of manual data re-entry.

**Why customers care.** This is the "full-stack automation" layer that most SMBs approach via Zapier + Make.com + brittle scripts. Post-`leger bootstrap automation`, every automation is version-controlled in the customer's infra repo, deployed via CI, observable via one dashboard, and durable (survives restarts). Most critically: **every automation writes to `run_ledger.db` by the same schema as everything else**, so the customer has one audit log across the entire stack.

## 7. Summary table — six verbs, 17+ surfaces, one config envelope

| Verb | Surfaces wired | Customer plan needed | Sprint readiness |
|---|---|---|---|
| `leger bootstrap perimeter` | Tunnel, Access, DNS, Turnstile | Free Zero Trust | Mostly existing; extend to Turnstile in sprint |
| `leger bootstrap dashboards` | Pages/Workers-with-static-assets, Access, Workers Builds | Free + Workers Paid at scale | Scaffolded in sprint; dashboards flesh out post-sprint |
| `leger bootstrap media` | R2, DNS, optionally Images | Free | R2 backup already; asset CDN post-sprint |
| `leger bootstrap identity` | Access IdP binding, Access for SaaS, Secrets Store | Free | Core identity binding in sprint; Access for SaaS post-sprint |
| `leger bootstrap ai` | AI Gateway, Workers AI, Analytics Engine | Workers Paid ($5/mo) recommended | AI Gateway + local-escalation path in sprint |
| `leger bootstrap automation` | Workers, Workers Builds, DO, Queues, Workflows, Observability | Workers Paid ($5/mo) | Sync Workers + DO state in sprint; Workflows post-sprint |

**The whole stack costs the customer $0–$15/month on Cloudflare**, depending on volume — and saves them materially more than that on LLM spend alone via AI Gateway caching, not counting the value of every other capability.

**The `leger.config.yaml` envelope grows to carry Cloudflare state.** A new top-level section:
```yaml
cloudflare:
  account_id: "..."   # customer's account ID
  zone_id: "..."      # customer's DNS zone
  features:
    perimeter: {enabled: true, turnstile: true}
    dashboards: {enabled: true, pages_project: "sodimo-dashboards"}
    media: {enabled: true, backup_bucket: "sodimo-backup"}
    identity: {enabled: true, idp: "google-workspace", saas_apps: ["twenty", "paperclip"]}
    ai: {enabled: true, gateway_id: "...", include_workers_ai: true}
    automation: {enabled: true, workers_builds: true, workflows: false}
```

This envelope is the deterministic converger input. `leger bootstrap <verb>` reads this, computes the diff vs actual CF account state, emits a PR to the customer's infra repo. After merge, the next `leger apply` re-runs `bootstrap` against the final state for verification + emits a summary `run_ledger` row.
