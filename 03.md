## Cross-cutting A: Bundle-deal posture?

Cloudflare's pricing is **not** formally bundled — there is no "adopt 10+ products, get X% off" meter. The two explicit bundles are **Workers Paid** ($5/mo base, Standard usage model) and **Cloudflare for Startups** credit tiers ($5K / $25K / $100K / $250K), the top two of which require VC backing or Launchpad cohort membership. Per the founder interview paraphrased in `noon-ecosystem-support-map.md`, Launchpad selection is *signal*-driven: "the newer and fresher/beta-er the better," and Leger at 15 integrated products (current 8 + 7 sprint-adjacent/post-sprint) is credibly in the zone that Cohort-6 alumni occupied at selection. The M&A channel is real — Nefeli (Cohort 2) and Outerbase (Cohort 4) were acquired — but skewed toward ex-Cloudflare founders per the same interview. There is no structural reason 15+ surfaces changes pricing, but there is a strong structural reason it changes selection odds: it makes the "meaningfully uses Workers" evaluation trivially pass, freeing the reviewer to focus on the actual product thesis rather than platform-fit. The implicit bundle deal, then, is "15 products = the benefit of the doubt" — Leger's goal.

## Cross-cutting C: Any CF product that would subsume `leger-release-manifest`?

Short answer: **no, not yet**. Long answer:

- **Artifacts** (mentioned in the Agents Week product lineup and `workflows product page`) is git-native versioned storage — positioned as a platform primitive for *runtime* versioned blobs, not release-manifest pinning. Different problem.
- **Workers Builds** (CI/CD integrated with git repos, `developers.cloudflare.com/workers/ci-cd/builds`) is a build pipeline, not a release manifest — it ships the *output* of a build, not a pinned tuple across artifacts.
- **Wrangler `wrangler deploy --secrets-file`** is now mature for per-Worker secrets but doesn't address cross-repo version tuples.
- **Secrets Store** comes closest as a home for the signing key, but doesn't hold the version tuple itself (which is a YAML file, not a secret).
- **Cloudflare is not building a "release manager" product.** Agents Week shipped compute (Dynamic Workers, Sandboxes), identity (Managed OAuth, scannable tokens), networking (Mesh, Workers VPC), observability (Workers Observability), AI (unified inference). None of these is "here's your supply-chain version catalog."

**Conclusion.** `leger-labs/leger-release-manifest` stays as-is — a git-tracked repo with cosign-signed YAML, promoted via Workflows (post-sprint adoption), with keys in Secrets Store. The closest CF analogue is the internal `cloudflare/skills` pattern (skills bundled as versioned git refs), which is not a product.

## Cross-cutting D: AI Gateway vs Workers AI vs Anthropic-direct — decision tree

The trichotomy resolves cleanly given Leger's Path (c) doctrine ("Claude Code optional, never load-bearing"):

```
Is the call happening on the customer's Framework Desktop (on-box LiteLLM quadlet)?
├── YES → Local llama.cpp first (LP-07). 
│         Only if local unavailable: egress via Tunnel → AI Gateway (Universal Endpoint) 
│         → provider of choice with per-request Vaultwarden-minted token.
└── NO (call is from app.leger.run Worker or leger.run telemetry worker)
    ├── Is the use-case schema-validation / prose generation for the configurator?
    │   └── Route via AI Gateway → Workers AI (Kimi K2.5 or Llama) by default; 
    │       fallback chain: Anthropic → OpenAI.
    ├── Is the use-case runbook-narration (Path c optional)?
    │   └── Customer uses their own Claude Code install; Leger does not proxy.
    └── Is the use-case documentation generation for leger-docs?
        └── AI Gateway → Workers AI (cheapest, same-network).
```

**Three principles:**
1. **AI Gateway is always in the path** for any cloud call — provider-agnostic, token-scoped, cost-tracked.
2. **Anthropic-direct is never the default** — always via AI Gateway so cf-aig-metadata tracks cost.
3. **Workers AI is the cheapest default provider** for internal-use Leger calls (docs, configurator validation, telemetry summaries) — no cross-cloud hops, neurons-based pricing is bounded and predictable.

For Path (c) doctrine: the customer's Claude Code install talks to *their own* Anthropic account, *not* via `app.leger.run`. Leger never proxies customer LLM calls — that would be a long-lived credential by the back door.

## Cross-cutting E: Queues / Workflows / Durable Objects — when to pick which

Cloudflare's own guidance is informal but consistent across the docs:

- **Durable Objects** — strongly-consistent state at a named address. Use when you need per-entity in-memory + on-disk state that survives requests and is coordinated (one writer, many readers or many writers serialized). Example: OAuth device-grant session.
- **Queues** — async message passing with delivery guarantees (at-least-once). Use when producer and consumer are decoupled, order matters, and failures should retry. Example: `email_outbox` (already used in LP-04/LP-15).
- **Workflows** — long-running multi-step orchestration with per-step retries and durable state. Use when you have a sequence of operations that must complete eventually and each step is independently retryable. Example: release-manifest promotion pipeline.

**Leger's specific application:**
- Per-engagement configurator draft state + OAuth device-grant window → **DO** (currently D1; migrate per row #5 above).
- `email_outbox`, future `backup_events` queue, future `runbook_progress` queue → **Queues** (already in use).
- Release-manifest promotion, nightly rotation-evidence aggregation, cosign key rotation ceremony → **Workflows** (post-sprint adoption).
- On-box `leger apply` / `leger rotate` / `leger bootstrap` runbooks → **neither — CLI on-box**, per Path (c) / survivability matrix.

**The rule of thumb from `durable-execution` docs:** "For work that is likely to outlive a single DO lifetime, fire-and-forget with checkpoint/recovery is the safer pattern" — which corresponds to Workflows. For short-lived work that needs strong consistency with a caller, use DOs and `runFiber()` for in-request resumability.

## Cross-cutting F: Observability story — minimum viable tenant-facing

What Leger should wire through on the **vendor side** (app.leger.run, telemetry worker, manifest-promoter):

1. **Workers Observability** (`observability = true` in wrangler) — free, always on, Query Builder for structured logs. Head-based sampling at 1.0 for alpha, drop to 0.1 once volume grows.
2. **Logpush → R2** — Workers Paid. Long-term retention at pennies per GB for compliance-evidence purposes (if a customer ever disputes a configurator change timestamp).
3. **Tail Worker → Analytics Engine** — custom sampler that writes per-engagement metrics (requests, p90 latency, error rate) into an `app_leger_run_per_engagement` dataset, queryable via SQL for FDE debugging.
4. **OTLP export (optional)** — per `workers/observability` docs, exports to Honeycomb, Grafana Cloud, Axiom, Sentry. Leger doesn't need this for v1 — Workers Observability + Logpush-to-R2 is enough.

On the **customer side** (on-box), observability is already covered:
- `run_ledger.db` is the append-only event log.
- Datasette quadlet (per `05-changelog-architecture.md` §6) is the tenant's interactive SQL browser.
- No cross-cloud log shipping — LP-04 forbids inbound ports.

**Minimum-viable tenant-facing surface:** `<customer>.leger.run/changelog` (already planned) + Datasette UI (already planned). No additional CF surface needed on the tenant side. Anything more is vendor-observability for Tom's debugging, not customer-observability for Michel's use.

## Cross-cutting G: M&A / competitive framing

Two Leger design areas overlap with active Cloudflare roadmap items. The Launchpad application should frame each as complementary:

**Workflows vs `leger bootstrap`/`leger rotate` runbook engine.** The overlap is shallow — Workflows is vendor-side orchestration; Leger's CLI verbs run on-box. Frame as: "Leger's CLI runs the on-box side of runbooks so the customer's stack is survivable without Cloudflare; Workflows is how Leger's own release ceremony stays durable when Tom is asleep." Complementarity angle: Leger's release-manifest promoter is a Workflows poster-child because it's exactly the human-in-the-loop + multi-step + can-sleep-for-days pattern Workflows was redesigned for.

**AI Gateway vs LiteLLM-in-cloud-escalation.** LiteLLM on-box stays primary (LP-07). AI Gateway is the *vendor side* fallback — when `app.leger.run` itself needs to call an LLM (schema validation, docs gen, telemetry summarization) it goes via AI Gateway. Frame as: "Leger treats AI Gateway as the cloud-side sibling of LiteLLM — identical OpenAI-compatible shape, identical provider-agnostic routing, but LiteLLM runs where the data is (on-box) and AI Gateway runs where the control plane is."

**Enterprise MCP vs per-engagement MCP Worker.** The overlap is aspirational — Cloudflare's internal MCP architecture (`internal-ai-engineering-stack`) uses MCP Server Portals + Code Mode to consolidate 13 MCP servers into one governed endpoint. Leger's equivalent is "one MCP Worker per engagement, gated by CF Access" — same architectural shape, different unit of tenancy (per-customer rather than per-employee). Frame as: "Leger is the SMB-scale version of what Cloudflare runs internally; same primitives, tenant-unit sized for 30-person operations."

**No surface is a head-to-head competitor.** Leger is the *deployment layer* for on-prem AI, running on *one Framework Desktop per customer*. Cloudflare is the *cloud-native serverless platform*. The overlap is vocabulary (MCP, Workflows, AI Gateway) not substance. Lead with that.

## Three questions answerable only by a CF human (Solutions Architect office hours first call)

1. **"What's the Secrets Store exit-from-beta roadmap?"** Currently one store per account, 100 secrets per account (secrets-store Wrangler docs, March 2026). Leger's manifest-promotion workflow + configurator + telemetry worker + docs-build worker will share ~5–10 secrets, which is comfortable inside 100, but if Leger grows to 20 customers and each has an engagement-scoped credential reference in the store, 100 gets tight fast. A definitive GA date + limit roadmap would let Leger plan for sub-accounts vs per-account expansion.

2. **"Is AI Gateway Unified Billing available on Workers Paid, or Enterprise-only?"** The Cloudflare AI Gateway Pricing Explained (truefoundry, April 2026) mentions "Unified Billing (New for 2026): Cloudflare now allows you to pay for third-party model usage (OpenAI, etc.) directly through your Cloudflare invoice." This would eliminate one more long-lived vendor credential (the OpenAI/Anthropic direct API token) for Leger's configurator cloud-escalation path. If Unified Billing is Workers Paid-compatible, Leger adopts it immediately; if Enterprise-only, Leger stays with per-request token minting from customer Vaultwarden.

3. **"Is Workers for Platforms dispatch-namespace access available without Enterprise?"** (This is a defensive question, not an adoption question.) Leger has explicitly *rejected* Workers for Platforms because Leger is not a multi-tenant-SaaS-hosting-customer-code. But if an FDE-network model ever emerges where FDEs host their own customers' configurator Workers, dispatch namespaces become relevant. Today the docs imply self-serve access on Workers Paid (`cloudflare-for-platforms/workers-for-platforms/reference/pricing` shows same request pricing as standard Workers) but historical Cohort interviews imply an Enterprise gate. Clarity would let Leger design the future "deployable by any FDE" story without locking into an Enterprise-tier dependency.
