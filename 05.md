# Leger × Cloudflare — customer-enablement investigation
## Part 1 of 3: Executive summary, mental model, decision matrix

**Prepared:** 2026-04-23
**Scope:** For every publicly-documented, at-least-open-beta Cloudflare surface as of April 2026, what can **the customer do with their own Cloudflare account**, and how does Leger's CLI + configurator + Claude Code steering automate that surface's setup?

**Reframe note.** The original investigation framing was "what should Leger-the-vendor use CF for?" That was wrong. The larger and more honest question is: **what does Leger-the-product enable its customers to do on their own Cloudflare accounts**, using the `leger` CLI as the deterministic setup wrapper and Claude Code as the steering layer? Leger's unit is one Framework Desktop per customer. The customer's Cloudflare account is the customer-owned edge that wraps that box. Leger's job is the declarative layer that wires the two together. This turns Leger from "a SaaS built on Cloudflare" into **a distribution channel for Cloudflare into SMB on-prem** — materially stronger Launchpad narrative.

---

## 1. Executive summary

Leger's customers own their Cloudflare accounts, their DNS zones, their Access tenants, their Workers, their R2 buckets, their Turnstile keys. Leger provides a `leger bootstrap cloudflare` family of CLI verbs that, given a customer API token with scoped permissions, idempotently provisions the full stack the LP-XX pack assumes — tunnel, Access applications, per-service hostnames, DNS records, R2 backup bucket, Turnstile widgets, AI Gateway instance, per-engagement Pages dashboards, and so on. The CLI runs in two modes: `--interactive` (Claude Code drives it via the MCP Worker, FDE approves each step) and `--declarative` (reads `leger.config.yaml` and converges idempotently, for re-runs and rotations).

**Six bootstrap verbs, each wrapping a cluster of CF surfaces:**

1. **`leger bootstrap perimeter`** — Tunnel + Access apps + Access policies + per-service DNS + Turnstile on public forms. This is the LP-01 + LP-04 core, already partially wired.
2. **`leger bootstrap dashboards`** — Pages-deployed (migrating to Workers-with-static-assets) internal team dashboards behind Access. Claude Code authors the dashboards from `run_ledger` schemas and pushes to the customer's infra repo; Workers Builds deploys on merge; Access policies wrap them.
3. **`leger bootstrap media`** — R2 age-encrypted backup bucket (LP-13), optional per-customer asset CDN for the tenant changelog.
4. **`leger bootstrap identity`** — Access as SAML/OIDC broker for the customer's Google Workspace (or other IdP); Access for SaaS to front their Twenty / Paperclip / Salesforce; Secrets Store for service tokens.
5. **`leger bootstrap ai`** — One AI Gateway instance per engagement, logging every LLM call outbound from the box; Workers AI as cloud escalation for Kimi K2.5 / Llama Guard; Analytics Engine for per-employee token-spend attribution.
6. **`leger bootstrap automation`** — Workers Builds CI for the `<customer>/infra` repo; Durable Objects for live dashboard state; Queues for the existing `email_outbox`; Workflows for customer business processes like quarterly rotation drills.

**Three Cloudflare surfaces reshape the product thesis entirely**, not just incrementally:

- **Cloudflare Access is the customer's IdP broker.** Free Zero Trust plan supports 50 users, 50 applications, 10GB transfer (developers.cloudflare.com/cloudflare-one/, 1 week ago) — comfortably enough for a 30-person SMB like Sodimo. Every Leger engagement gets Access as the front door to every on-box service AND every cloud SaaS the customer buys. That is a strong, self-serve SMB story that almost nobody is telling outside the tech-first segment.

- **Cloudflare Pages (→ Workers-with-static-assets) is the natural home for per-engagement internal dashboards.** The customer has ~6 team members per Sodimo-like engagement who need "what's my stock level", "how's my token spend this month", "when was the last backup?" dashboards. These dashboards render off `run_ledger.db` (queried via a customer-owned Worker that reaches the box via Tunnel), deploy to the customer's own Pages, and are gated by Access policies — total cost: $0 on Free Zero Trust + Pages free. Leger-Labs does not host them; the *customer's* CF account does. This is load-bearing.

- **The Cloudflare Code Mode MCP server (`mcp.cloudflare.com`) wires Claude Code into the customer's CF account directly.** Two tools (`search()`, `execute()`), ~1,000 tokens total, covers 2,500+ Cloudflare API endpoints via OAuth-scoped permissions (blog.cloudflare.com/code-mode-mcp, March 9, 2026 and github.com/cloudflare/mcp, 6 days ago). The customer's Claude Code installs this MCP once, and from then on Claude can execute any Cloudflare operation against the customer's own account — no Leger proxy, no long-lived vendor credential. Combined with the `leger` MCP Worker (already planned), Claude Code sees **two MCPs at once**: `leger` for the box, `cloudflare` for the cloud edge. A runbook like "add a new Piler service behind Access with Google SSO and a DNS record" becomes one Claude Code conversation spanning both.

**Claude Code's role becomes concrete.** The customer's Claude Code install has three ways to drive infrastructure change:
1. **Direct via Cloudflare MCP** — "configure an Access application for piler.sodimo.eu with the policy engineering@sodimo.eu" → Claude executes the API call directly.
2. **Indirect via `leger` CLI MCP Worker** — "add Piler as a new service" → Claude calls `leger configure service add piler` which emits a `leger.config.yaml` diff, which the configurator's PR-to-infra flow formalizes, which `chezmoi apply` renders on-box.
3. **Hybrid via both** — "add Piler end-to-end": Claude calls `leger configure service add piler` (updates config), then `mcp.cloudflare.com/execute` (creates the CF Access app), then `leger doctor` (validates the setup).

Path 3 is the runbook. Claude Code steers both MCPs in one session; every step writes into `run_ledger.db` as `surface='claude-code', actor='<fde-name>', change_type=<verb>`. The FDE approves each step. The whole session is the release-artifact.

**Plan-tier detection is new CLI scope.** `leger doctor --cloudflare-features` reads the customer's CF account plan and reports what's unlocked. Access Free (50 users) is fine for Sodimo (30 people); Workers Paid ($5/mo) unlocks Durable Objects key-value storage + higher Logs volume + Queues. Workers Free is enough for v1 for most customers. The CLI gates features accordingly — `leger bootstrap perimeter --enable-workers-ai` requires Workers Paid and the CLI says so clearly. **Leger never encourages customer upgrades without showing the specific feature the upgrade unlocks and the dollar threshold above which it pays for itself.**

**The Workers Launchpad narrative pivots completely.** Leger is not "a SaaS built on Cloudflare" — Leger is "a packaging and steering layer that brings the Cloudflare developer platform to SMBs on their own account." That is a *much* stronger pitch for selection. Precedent: Cloudflare has always said that Workers Launchpad favours companies "meaningfully using Workers" — Leger's pitch is "we drive five-figure MRR of Workers adoption into SMB accounts, accounts that would not otherwise reach the Cloudflare platform." The acquisition thesis for Cloudflare is also stronger on this framing: Nefeli and Outerbase were acquired as infrastructure primitives; Leger's framing is "go-to-market primitive — we reach a segment Cloudflare's direct sales cannot touch economically."

**Three loops.**

**Loop 1: configurator → infra repo → box.** `app.leger.run` emits a PR to `<customer>/infra`; customer merges; `chezmoi apply` on the box renders the new state. (Already designed per `04-pr-to-infra-output.md`.)

**Loop 2: `leger bootstrap` → Cloudflare API → customer's CF account.** The CLI on the box (or run locally by the FDE during an initial setup), using a scoped customer API token, idempotently converges CF resources via the Cloudflare API (optionally using the Terraform provider under the hood). Results write back into `leger.config.yaml` as state references, then the PR-based loop takes over.

**Loop 3: Claude Code → two MCPs → both sides at once.** Claude Code, running on the FDE's machine or directly on the box, has `leger` MCP + `cloudflare` MCP both configured. A single conversation can span "what's on the box" and "what's on the cloud edge," and every step emits an entry to `run_ledger` for audit.

The three loops are compositional: a Claude Code session (Loop 3) triggers a `leger configure` step (Loop 1) and a `leger bootstrap cloudflare` step (Loop 2), and all three write their events into `run_ledger.db` with identical schema.

## 4. Summary count

- **Customer-facing core surfaces (rows 1–8):** 8, mostly already in use via LP-01 + LP-04. Live on customer's CF Free plan.
- **Adopt now (rows 9–19):** 11 new surfaces wrapped by the 6 `leger bootstrap` verbs. Most sit inside customer's Free + Workers Paid ($5/mo) tier.
- **Adopt post-sprint (rows 20–22):** 3 surfaces.
- **Evaluate later / reject (rows 23–33):** 11 surfaces, each with rationale.

**Total customer-facing CF surfaces per engagement, when fully bootstrapped: 17–19.** On a typical customer Workers Paid ($5/mo) plan, this costs the customer roughly $5–15/month on Cloudflare depending on volume — a material markup on zero, but tiny relative to what the customer saves on the compute they no longer buy from a cloud vendor (`run_ledger`'s `cost_eur_if_cloud` column).

**Leger's own CF surfaces (vendor-side, distinct from this investigation):** `app.leger.run` runs on Leger's CF account as the only vendor-hosted surface. Currently 5 surfaces (Workers + Pages + D1 + R2 + Tunnel). That's the first-pass investigation's scope — largely unchanged, trivially expanded to include Secrets Store (for the GH App private key) and Workflows (for release-manifest promotion).
